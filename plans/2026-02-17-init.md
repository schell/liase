# liase — Initial Architecture Plan

**Date:** 2026-02-17
**Author:** schell + claude-opus-4-6
**Status:** Draft

## What Is liase?

**liase** is a Tauri v2 desktop application for developer liaisons who need to monitor
and respond across many communication platforms from a single unified inbox. Built with
a Rust backend and a mogwai (Rust-to-WASM) frontend, following the architectural
patterns established in winatep/DoAsIf.

The name is a portmanteau of "liaison" + "liaise".

## Problem Statement

A Zcash developer liaison must monitor and respond across:

- **GitHub** — issues, PRs, discussions, notifications across multiple Zcash-related repos
- **Discourse** — https://forum.zcashcommunity.com/
- **Discord** — Zcash community server(s)
- **Slack** — internal/team channels
- **Signal** — private conversations
- **Facebook** — community pages and messenger

No existing tool covers all six platforms with a true unified inbox, two-way
communication, and AI integration. The closest options are:

- **Webview aggregators** (Ferdium, Rambox, Wavebox): cover most platforms as tabs,
  but cannot support Signal (no web client), provide no unified inbox, and have
  limited/no AI integration.
- **Beeper**: true unified inbox with Signal, Discord, Slack, and Facebook support,
  but no GitHub or Discourse integration.
- **Matrix + bridges**: can bridge Discord, Signal, Slack, and Facebook, but has no
  GitHub issues/PR support, no Discourse bridge, and requires complex self-hosting.
- **Community intelligence tools** (Common Room, Orbit): analytics-focused, not for
  daily message reading/responding.

There are **no existing Tauri-based unified messaging apps**.

## Design Decisions

### Technology Stack

| Layer | Choice | Rationale |
|---|---|---|
| Desktop framework | Tauri v2 | Native webview, Rust backend, small binary (~600KB vs Electron's 100MB+), sidecar support |
| Frontend framework | mogwai 0.7 (Rust-to-WASM) | Single-language stack, follows winatep patterns, `V: View` generic for testability |
| Frontend patterns | Capabilities traits, `step()` event loop, `rsx!`, `ViewChild`, `Proxy<T>` | Proven in winatep/DoAsIf, clean separation of concerns |
| Database | SQLite via rusqlite (bundled) | Local-first, zero config, WAL mode for concurrent reads, FTS5 for search |
| Secrets storage | tauri-plugin-stronghold | Encrypted at rest, purpose-built for Tauri |
| AI (initial) | Anthropic Claude via misanthropic crate | Best reasoning for summarization/drafting; architecture allows future swap to local models |
| AI (future) | Ollama / local models | Pluggable via `AiProvider` trait boundary |
| Styling | Bootstrap 5 + FontAwesome | Consistent with winatep, fast to develop |

### Platform Connection Methods

| Platform | Auth Method | Crate/Approach | ToS Risk |
|---|---|---|---|
| GitHub | Personal Access Token (fine-grained) | `octocrab` | None — officially supported |
| Discourse | API Key + Username header | `reqwest` (thin client) | None — officially supported |
| Discord | User token (raw HTTP + Gateway WebSocket) | `reqwest` | **HIGH** — self-bots are banned by ToS; account termination risk is real and enforced |
| Slack | User OAuth (`xoxp-*` token) + Socket Mode | `slack-morphism` | None — officially supported |
| Signal | Phone registration via signal-cli | signal-cli JSON-RPC over TCP (sidecar) | Low — unofficial but tolerated for 10+ years |
| Facebook | User OAuth + Page tokens (Graph API) | `reqwest` | Low — officially supported but requires App Review for non-basic permissions |

### Signal Sidecar

signal-cli requires a Java 21+ runtime. It will be bundled as a Tauri sidecar with a
minimal JRE created via `jlink` (~30MB). The sidecar runs in daemon mode over TCP,
communicating via JSON-RPC.

### Discord Risk Acceptance

Discord actively detects and bans self-bots. This risk is accepted with the following
mitigations:
- Rate limit aggressively (well below documented limits)
- Read-heavy usage patterns (minimize automated writes)
- No automated actions that look non-human (no mass operations)
- Keep a backup Discord account

### User Count

This is a single-user tool. No multi-user collaboration features are needed. All data
is local.

## Unified Message Model (wire-types)

Every platform's messages get normalized into a common type:

```rust
/// Globally unique message identifier.
pub struct MessageId {
    pub platform: Platform,
    pub native_id: String,
}

/// Which platform a message originates from.
pub enum Platform {
    GitHub,
    Discourse,
    Discord,
    Slack,
    Signal,
    Facebook,
}

/// Platform-specific context for where a message lives.
pub enum Channel {
    GitHubIssue { owner: String, repo: String, number: u64 },
    GitHubPR { owner: String, repo: String, number: u64 },
    GitHubDiscussion { owner: String, repo: String, number: u64 },
    GitHubNotification { id: String },
    DiscourseTopicPost { topic_id: u64, post_number: u64 },
    DiscordChannel { guild_id: u64, channel_id: u64 },
    DiscordDM { user_id: u64 },
    SlackChannel { team_id: String, channel_id: String },
    SlackDM { team_id: String, user_id: String },
    SignalChat { recipient: String },
    SignalGroup { group_id: String },
    FacebookPage { page_id: String },
    FacebookMessenger { conversation_id: String },
}

/// A message author.
pub struct Author {
    pub display_name: String,
    pub avatar_url: Option<String>,
    pub platform_identity: String, // @username, phone number, etc.
}

/// The content of a message.
pub struct MessageContent {
    pub text: String,                    // Plain text or markdown
    pub html: Option<String>,           // Pre-rendered HTML if available
    pub attachments: Vec<Attachment>,
}

pub struct Attachment {
    pub filename: String,
    pub url: String,
    pub mime_type: Option<String>,
    pub size_bytes: Option<u64>,
}

/// A normalized message from any platform.
pub struct Message {
    pub id: MessageId,
    pub platform: Platform,
    pub channel: Channel,
    pub thread_id: Option<ThreadId>,
    pub author: Author,
    pub content: MessageContent,
    pub timestamp: DateTime<Utc>,
    pub read: bool,
    pub starred: bool,
    pub metadata: PlatformMetadata,
}

/// Platform-specific metadata that doesn't fit the common model.
pub enum PlatformMetadata {
    GitHub {
        state: Option<String>,        // open, closed, merged
        labels: Vec<String>,
        is_notification: bool,
        notification_reason: Option<String>,
    },
    Discourse {
        category: Option<String>,
        tags: Vec<String>,
        like_count: u32,
    },
    Discord {
        reactions: Vec<Reaction>,
        edited: bool,
    },
    Slack {
        reactions: Vec<Reaction>,
        edited: bool,
        thread_reply_count: Option<u32>,
    },
    Signal {
        is_group: bool,
        receipt_status: Option<String>,
    },
    Facebook {
        is_page_post: bool,
        reaction_count: u32,
    },
}

pub struct Reaction {
    pub emoji: String,
    pub count: u32,
}

pub struct ThreadId {
    pub platform: Platform,
    pub native_id: String,
}
```

## Platform Adapter Trait

```rust
#[async_trait]
pub trait PlatformAdapter: Send + Sync {
    /// Which platform this adapter serves.
    fn platform(&self) -> Platform;

    /// Fetch new/updated messages since the given checkpoint.
    async fn poll(&self, since: Option<DateTime<Utc>>) -> Result<Vec<Message>>;

    /// Send a reply in the context of an existing message/thread.
    async fn reply(&self, thread: &ThreadId, content: &str) -> Result<Message>;

    /// Mark a message as read on the remote platform.
    async fn mark_read(&self, message_id: &MessageId) -> Result<()>;

    /// Get the full thread/conversation for a message.
    async fn get_thread(&self, thread_id: &ThreadId) -> Result<Vec<Message>>;

    /// Platform-specific health check (auth valid, API reachable).
    async fn health_check(&self) -> Result<PlatformStatus>;
}
```

### Adapter Implementation Notes

**GitHub** (`octocrab`):
- Poll `GET /notifications` (respects `X-Poll-Interval` header, typically 60s)
- List issues/PRs/comments on curated repos via `GET /repos/{owner}/{repo}/issues`
- Reply via `POST /repos/{owner}/{repo}/issues/{number}/comments`
- Fine-grained PAT scoped to specific repos

**Discourse** (`reqwest`):
- Poll `GET /notifications.json` + `GET /latest.json`
- Auth via `Api-Key` + `Api-Username` headers
- Reply via `POST /posts.json`
- ~200 lines of Rust for a thin client

**Discord** (`reqwest` + Gateway WebSocket):
- Connect to Gateway (`wss://gateway.discord.gg`) with user token
- Receive `MESSAGE_CREATE` and other events in real-time
- REST API for history: `GET /channels/{id}/messages`
- Reply via `POST /channels/{id}/messages`
- Set `$os`, `$browser` fields to mimic a real client
- Rate limit carefully (well below published limits)

**Slack** (`slack-morphism`):
- User OAuth flow (`xoxp-*` token) with Socket Mode
- Socket Mode provides real-time events without a public webhook URL
- Reply via `chat.postMessage` with user token
- Full two-way communication as the authenticated user

**Signal** (signal-cli JSON-RPC over TCP):
- Spawn signal-cli as Tauri sidecar: `signal-cli daemon --tcp localhost:7583`
- Connect via `TcpStream`, send/receive JSON-RPC
- Receive messages as JSON-RPC notifications
- Send via `{"method": "send", "params": {"recipient": ["+1..."], "message": "..."}}`
- Registration/linking handled through settings UI

**Facebook** (`reqwest` + Graph API):
- User OAuth flow -> long-lived token (60 days) -> Page Access Token
- Poll `GET /{page-id}/conversations` for Messenger
- Poll `GET /{page-id}/feed` for page posts
- Reply via `POST /{conversation-id}/messages` or `POST /{post-id}/comments`
- Requires Facebook App creation + App Review for non-basic permissions

## AI Provider Trait

```rust
#[async_trait]
pub trait AiProvider: Send + Sync {
    /// Summarize a thread of messages into a concise overview.
    async fn summarize(&self, messages: &[Message]) -> Result<String>;

    /// Draft a reply given thread context and optional user instructions.
    async fn draft_reply(
        &self,
        messages: &[Message],
        instructions: Option<&str>,
    ) -> Result<String>;
}
```

### Claude Implementation

The initial implementation uses the `misanthropic` crate for the Anthropic Messages API.
System prompts are tailored per action:

- **Summarize**: "You are summarizing a conversation from {platform} for a Zcash
  developer liaison. Focus on technical questions, decisions made, action items, and
  any blockers mentioned."
- **Draft reply**: "You are drafting a reply as a Zcash developer liaison. The reply
  should be technically accurate, helpful, and match the tone of the platform. The
  user will review and edit before sending."

The `AiProvider` trait is the abstraction boundary for future migration to local models
(Ollama, llama.cpp, etc.).

## Local Storage (SQLite)

All data is stored locally in a SQLite database with WAL mode enabled.

### Schema

```sql
CREATE TABLE messages (
    id TEXT PRIMARY KEY,           -- "{platform}:{native_id}"
    platform TEXT NOT NULL,
    channel_json TEXT NOT NULL,    -- JSON-serialized Channel enum
    thread_id TEXT,
    author_json TEXT NOT NULL,     -- JSON-serialized Author
    content_json TEXT NOT NULL,    -- JSON-serialized MessageContent
    timestamp TEXT NOT NULL,       -- ISO 8601
    read INTEGER DEFAULT 0,
    starred INTEGER DEFAULT 0,
    metadata_json TEXT,            -- JSON-serialized PlatformMetadata
    created_at TEXT DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE threads (
    id TEXT PRIMARY KEY,           -- "{platform}:{native_thread_id}"
    platform TEXT NOT NULL,
    title TEXT,
    last_message_at TEXT,
    unread_count INTEGER DEFAULT 0,
    metadata_json TEXT
);

CREATE TABLE platform_configs (
    platform TEXT PRIMARY KEY,
    config_json TEXT NOT NULL,     -- Non-secret config (secret refs -> stronghold)
    enabled INTEGER DEFAULT 1,
    last_poll_at TEXT,
    poll_interval_secs INTEGER DEFAULT 60
);

CREATE TABLE ai_interactions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    message_id TEXT REFERENCES messages(id),
    thread_id TEXT,
    action TEXT NOT NULL,          -- 'summarize', 'draft_reply'
    input_json TEXT,               -- What was sent to the AI
    output_text TEXT,              -- What came back
    model TEXT,
    created_at TEXT DEFAULT CURRENT_TIMESTAMP
);

-- Performance indexes
CREATE INDEX idx_messages_timestamp ON messages(timestamp DESC);
CREATE INDEX idx_messages_thread ON messages(thread_id);
CREATE INDEX idx_messages_unread ON messages(read, timestamp DESC);
CREATE INDEX idx_messages_platform ON messages(platform, timestamp DESC);

-- Full-text search across all messages
CREATE VIRTUAL TABLE messages_fts USING fts5(
    content, author_name, title,
    content='messages', content_rowid='rowid'
);
```

Credentials (API keys, tokens, passwords) are stored separately using
`tauri-plugin-stronghold` (encrypted at rest), **not** in SQLite.

## Backend Orchestrator (core)

```rust
pub struct Core {
    adapters: Vec<Box<dyn PlatformAdapter>>,
    store: Store,
    ai: Box<dyn AiProvider>,
}

impl Core {
    /// Start polling all enabled platforms. Each platform runs as a
    /// separate tokio task. New messages are written to the store and
    /// emitted as Tauri events.
    pub async fn start(&self, app_handle: tauri::AppHandle) -> Result<()>;

    /// Get unified inbox (paginated, filterable by platform/read/starred).
    pub async fn get_inbox(&self, filter: InboxFilter) -> Result<Vec<Message>>;

    /// Get full thread for a message.
    pub async fn get_thread(&self, thread_id: &ThreadId) -> Result<Vec<Message>>;

    /// Send reply via the appropriate platform adapter, store the result.
    pub async fn reply(&self, thread_id: &ThreadId, content: &str) -> Result<Message>;

    /// AI: summarize a thread.
    pub async fn summarize_thread(&self, thread_id: &ThreadId) -> Result<String>;

    /// AI: draft a reply.
    pub async fn draft_reply(
        &self,
        thread_id: &ThreadId,
        instructions: Option<&str>,
    ) -> Result<String>;

    /// Full-text search across all stored messages.
    pub async fn search(&self, query: &str) -> Result<Vec<Message>>;
}
```

Polling tasks run as `tokio::spawn` futures. Each task:
1. Calls `adapter.poll(since)` at the configured interval
2. Writes new messages to the store via `store.upsert_messages()`
3. Emits `app_handle.emit("new-messages", &new_messages)` for the frontend

## UI Architecture (mogwai)

### Patterns (from winatep)

All UI code follows these established patterns:

- **`V: View` generic** on all components (backend abstraction)
- **`#[derive(ViewChild)]`** with `#[child]` for DOM root elements
- **`Proxy<T>`** for reactive DOM bindings (class, text, attributes)
- **`async fn step(&mut self, caps: &mut impl XxxCapabilities)`** cooperative event loop
- **Capabilities traits** define side-effect interfaces (no Tauri/backend knowledge in ui crate)
- **`#[cfg(feature = "library")]`** sandbox modules with dummy capabilities for isolated development
- **`rsx!`** macro for declarative DOM construction
- **`EventListener`** async event streams consumed via `.next().await`
- **Event racing** via `FutureExt::or()` and `mogwai::future::race_all()`
- **`ProxyChild<V>`** for dynamic child swapping (pane content)
- **`localStorage`** persistence via `storage.rs`

## Implementation Phases

### Phase 1: Foundation (Weeks 1-2)

- Workspace scaffold (Cargo.toml, all crates with stubs)
- `wire-types`: Message, Platform, Channel, Author, ThreadId types with serde derives
- `store`: SQLite schema, migrations, basic CRUD (upsert_messages, get_inbox, get_thread)
- `ui`: App shell layout (three-pane), primitive components (port from winatep patterns)
  - Flavor, Alert, Button, Icon, List, Pane, Tab, Badge
  - AppShell with sidebar, inbox list, detail panel (all with dummy data)
- `frontend`: Tauri IPC wiring, dummy capabilities
- Tauri config, build pipeline (Trunk + Tauri)
- Component sandbox working (`trunk serve crates/ui/index.html`)

**Milestone:** App launches, shows three-pane layout with dummy messages.

### Phase 2: First Platform — GitHub (Weeks 3-4)

- `platforms::github`: Full adapter using octocrab
  - Poll notifications + specific repo issues/PRs/comments
  - Reply to issues/comments
  - Mark notifications as read
- `core`: Basic orchestrator (single platform polling, store integration)
- End-to-end flow: poll -> store -> display in inbox -> open thread -> reply
- Settings UI for GitHub PAT configuration

**Milestone:** Real GitHub notifications appear in the inbox. Can read threads and reply.

### Phase 3: Discourse (Week 5)

- `platforms::discourse`: REST API thin client for forum.zcashcommunity.com
  - Poll notifications + latest topics
  - Get topic threads with all posts
  - Reply to topics
- Settings UI for Discourse API key

**Milestone:** Zcash forum topics appear alongside GitHub in the unified inbox.

### Phase 4: Slack (Weeks 5-6)

- `platforms::slack`: slack-morphism with Socket Mode
  - User OAuth flow (in-app browser for redirect)
  - Real-time message reception via Socket Mode
  - Reply to channels and DMs
- Settings UI for Slack OAuth

**Milestone:** Slack messages flow in real-time. Can reply from liase.

### Phase 5: Discord (Weeks 6-7)

- `platforms::discord`: Gateway WebSocket + REST API
  - User token auth (manual entry in settings)
  - Gateway connection for real-time events
  - REST for message history
  - Reply to channels and DMs
- Rate limiting infrastructure

**Milestone:** Discord messages appear in inbox. Can read and reply.

### Phase 6: AI Integration (Weeks 7-8)

- `ai::claude`: Summarize and draft-reply via Anthropic API
- AI panel component in thread detail view
  - "Summarize" button -> displays summary above thread
  - "Draft Reply" button -> populates compose box with editable draft
  - Optional instructions input for draft guidance
- Settings UI for Anthropic API key
- `ai_interactions` table for history/caching

**Milestone:** Can summarize any thread and get AI-drafted replies.

### Phase 7: Signal (Weeks 8-9)

- signal-cli sidecar bundling (Tauri externalBin + minimal JRE via jlink)
- `platforms::signal`: JSON-RPC adapter over TCP
  - Daemon lifecycle management (start/stop)
  - Phone registration/linking flow in settings
  - Receive individual and group messages
  - Send replies
- Settings UI for Signal phone number + verification

**Milestone:** Signal messages appear in inbox. Can reply from liase.

### Phase 8: Facebook (Weeks 9-10)

- `platforms::facebook`: Graph API adapter
  - User OAuth flow (in-app browser)
  - Long-lived token management (auto-refresh before 60-day expiry)
  - Page conversations (Messenger) + page post comments
  - Reply to conversations and comments
- Settings UI for Facebook OAuth + page selection

**Milestone:** Facebook page messages and comments appear in inbox.

### Phase 9: Polish (Weeks 10-12)

- Full-text search (FTS5) with search UI
- Keyboard shortcuts (j/k navigation, r to reply, s to star, etc.)
- Native OS notifications via tauri-plugin-notification
- Unread count badges in sidebar per platform
- Performance tuning:
  - Message list virtualization for large inboxes
  - Lazy thread loading
  - Efficient polling intervals (backoff when idle)
- Error handling:
  - Retry logic with exponential backoff for failed API calls
  - Offline resilience (queue replies, send when back online)
  - Connection status indicators per platform
- Platform status indicators in sidebar (connected/disconnected/error)

**Milestone:** Production-ready single-user desktop app.

## Risks and Mitigations

| Risk | Severity | Mitigation |
|---|---|---|
| Discord account ban (self-bot ToS violation) | High | Rate limit aggressively, read-heavy usage, backup account |
| signal-cli + JRE bundle size (~30-100MB) | Medium | Use `jlink` for minimal JRE (~30MB), compress sidecar |
| Facebook App Review rejection | Medium | Start with basic page reading, apply for permissions incrementally |
| mogwai ecosystem (smaller than React/Svelte) | Medium | Markdown rendering via pulldown-cmark + `web-sys` innerHTML; custom components where needed |
| Platform API changes / deprecations | Low | Each adapter is isolated; API changes affect only one module |
| misanthropic crate maintenance | Low | Anthropic API is simple (~3 endpoints); can fall back to raw reqwest if needed |
| signal-cli breaking changes | Low | Pin signal-cli version in sidecar; update deliberately |
